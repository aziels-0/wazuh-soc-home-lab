# Custom Rule Detection Report

## Objective

Create a custom Wazuh detection rule for a simulated suspicious administrator action and map the alert to MITRE ATT&CK.

## Environment

- Wazuh server: `wazuh-server`
- Wazuh version: `4.14.6`
- Endpoint: `ubuntu-endpoint-01`
- Agent ID: `001`
- Endpoint IP: `192.168.64.4`
- Endpoint OS: `Ubuntu 26.04`
- Custom log file: `/var/log/soc-lab.log`

## Endpoint Log Collection Configuration

The endpoint agent was configured to collect a custom local log file:

```xml
<localfile>
  <log_format>syslog</log_format>
  <location>/var/log/soc-lab.log</location>
</localfile>
```

After updating the endpoint configuration, the log file was created and the Wazuh agent was restarted.

```bash
sudo touch /var/log/soc-lab.log
sudo systemctl restart wazuh-agent
```

## Custom Wazuh Rule

The following custom rule was added on the Wazuh server:

```xml
<group name="soc_lab,">
  <rule id="100100" level="12">
    <match>SOC_LAB_TEST suspicious_admin_action</match>
    <description>Potential credential-access activity detected in SOC lab</description>
    <mitre>
      <id>T1003</id>
    </mitre>
  </rule>
</group>
```

The Wazuh manager was restarted after adding the rule.

```bash
sudo systemctl restart wazuh-manager
```

## Test Performed

I wrote a simulated suspicious event into the custom log file on the endpoint.

```bash
echo "SOC_LAB_TEST suspicious_admin_action user=labuser action=credential_dump_attempt" | sudo tee -a /var/log/soc-lab.log
```

## Wazuh Detection

- Dashboard section: Threat Hunting
- Query used: `rule.id:100100`
- Agent: `ubuntu-endpoint-01`
- Rule ID: `100100`
- Rule level: `12`
- Rule group: `soc_lab`
- Rule description: `Potential credential-access activity detected in SOC lab`
- MITRE ATT&CK ID: `T1003`
- MITRE tactic: `Credential Access`
- MITRE technique: `OS Credential Dumping`

## Analysis

The custom Wazuh rule successfully detected the simulated suspicious administrator action from the endpoint's custom log file. This demonstrates that Wazuh can be extended beyond default rules by creating custom detections for organization-specific activity.

Mapping the rule to MITRE ATT&CK improves the alert context by connecting the event to a known adversary behavior: OS Credential Dumping. In a real environment, this type of detection could help identify attempts to access password hashes, credential stores, or other sensitive authentication material.

## Analyst Response

An analyst reviewing this alert should:

- Confirm whether the event was generated by an authorized test or real activity.
- Review the user account listed in the event.
- Search for related credential access or privilege escalation alerts.
- Check recent process execution, authentication activity, and file changes on the endpoint.
- Escalate the alert if the activity cannot be explained by approved administrative work.
